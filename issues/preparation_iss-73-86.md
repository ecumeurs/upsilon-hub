## 1. Context

The user wants to ship three issues in one overnight run because they form a single dependency chain:

1. **ISS-085** *(structural)* — extract `battlearena/property/` and `entity/skill/skillweight/` into a new shared Go module `github.com/ecumeurs/upsilon-mechanics` (repo `git@github.com:ecumeurs/upsilon-mechanics.git`, already provisioned by user 2026-04-26). This decouples "combat math" from "battle execution" so future generation services (skillgenerator-as-binary, region/token-based skill crafting) can depend on math without dragging the whole engine.

2. **ISS-086** *(data layer)* — promote items and skills from hardcoded/seeded content to admin-managed registries. `shop_items` becomes the live catalog (CRUD via custom Inertia admin); a new `skill_templates` table provides the skill design library. Designers gain a content-management surface without code changes.

3. **ISS-073** *(player-facing)* — character skill inventory + slot-based equipment. Characters acquire skills (currently only via the in-engine generator endpoint; shop/random drop wired later), store them in inventory, and equip up to `min(5, 1 + floor(wins/10))` for battle. Roguelike strategic-loadout depth.

**Why the chain matters:**
- ISS-073 needs an `equipped_skills` payload in `ArenaStartRequest` carrying full skill JSON (Laravel-master decision, see D2). That payload structure must mirror `equipped_items` from ISS-074, which currently lives in `upsilonapi/api/input.go`. If ISS-085 moves property types underneath it, the bridge resolver in ISS-073 lands cleanly on the new module's symbols on first write — no migration churn.
- ISS-086's skill_templates seed data is generated by calling the existing `skillgenerator` (Go) — which depends on `skillweight` — once. After ISS-085, both live in the shared module so a future "admin generates a skill via the UI" flow can call the same package from any Go service (or a CLI seed script).
- ISS-073's acquisition endpoint needs to call the generator too. Same logic.

**Foundations already in place** (from ISS-074, do not re-invent):
- `users.credits` ledger + `credit_transactions` audit table.
- `shop_items`, `player_inventory`, `inventory_transactions`, `character_equipment` tables.
- `EquippedItem` payload + `Forever=true` buff registration in `bridge.go`.
- `Entity.RegisterSkill(s skill.Skill)` at `upsilonbattle/battlearena/entity/entity.go:186-188` — already the registration call to use for skills.
- `Entity.EquippedSkills []string` field at `upsilonapi/api/input.go:34` with `// reserved for ISS-073` comment — **will be replaced** with full struct payload (see Phase 6).
- Custom Inertia admin (no Filament/Nova/Backpack); `User::isAdmin()` gate; existing `Admin/{Dashboard,UserManagement,History}.vue` pages and `AdminController` API endpoints.

**Skill engine state** (from explore):
- `Skill` struct at `upsilonbattle/battlearena/entity/skill/skill.go:10-20` — fields: `ID, Name, Behavior, Targeting, Costs, Effect, Cooldown`. No grade/weight stored on the struct (computed from properties).
- `skillgenerator.GenerateRandomSkill()` returns a balanced skill (uses `skillweight.Calculate` internally).
- `skillweight.Calculate(s) → (positive, negative, net)` and `skillweight.GetGrade(positiveSW) → "I".."V"`, `skillweight.GetCreditCost(positiveSW) → positiveSW * 2`.
- **No in-memory engine-side skill registry exists.** This is by design per D2: Laravel is the master, payload carries everything.

## 6. Adjacent findings & follow-ups

- **`EffectProperty` does NOT carry skill UUIDs.** The ISS-074 prep doc and ISS-073 wiring instructions both incorrectly imply weapon-as-skill is wired. Fix the misleading sentence in ISS-073 (Phase 0). With D11 in place, **exotic weapons can now grant a skill as a content concern** (admin attaches a `skill_template_id` to the shop_item). The original "weapon-as-skill" V2.1 framing is partially dissolved: vanilla weapons stay on the existing stat-property path (no skill needed — engine's `Attack()` reads `WeaponBaseDamage` directly), exotic weapons (grenade launcher etc.) layer a skill on top via D11. Throwables and traps use the same D11 path.
- **`skillgenerator` not extracted in ISS-085.** Documented as D10. New follow-up `ISS-090` planned to extract `Skill` struct and `skillgenerator` to `upsilon-mechanics` once the engine no longer needs the struct directly (or once we accept the upsilon-mechanics dependency on a thin skill-types subpackage).
- ~~**Per-player vs per-character `wins` source for skill_slots accessor.**~~ → **Resolved 2026-04-26: per-player (`users.total_wins`).** No `characters.wins` column needed.
- **Random shop (ISS-089).** Not in scope. The chosen registry shape (D3) supports it directly: `shop_items WHERE available=true` is the sample pool; deterministic seed (user UUID + creation date + current date) drives selection. ISS-089 can ship without schema changes.
- **Reforging.** Out of scope per user. `character_skills.instance_data` is JSON-snapshot specifically to make reforging straightforward later (mutate the snapshot in place).
- **Skill payload property typing.** Sending `map[string]interface{}` is pragmatic but loses type safety on the wire. V2.1 candidate: typed property serialization (`{"key": "Damage", "type": "int", "value": 50}`). Not needed for V2.0 since the engine has the same property registry.
- **Property factory mapping in bridge.** The `buildPropertyMap` and `buildEffect` helpers in `bridge.go` need to map every property key the skill JSON might carry. Items (ISS-074) had a small fixed set (ArmorRating, WeaponBaseDamage, etc.); skills have a much broader surface (Damage, Heal, ShieldPower, Stun%, Poison%, AoE, Range, Cooldown, MPLeech, SPLeech, HPLeech, Channeling, Delay, Accuracy, CritChance, etc.). Phase 6 implementation should table-drive this mapping for maintainability.
